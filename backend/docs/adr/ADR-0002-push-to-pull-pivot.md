# ADR-0002: 从批量推送改为按需拉取

## 状态

已接受

## 背景

v0.1.0 的核心流程是：用户提交问卷 → 系统一次性生成 5 个脚本 → 用户从列表中选择。

实际使用中发现以下问题：
1. **相关性不足**：5 个脚本是一次性批量生成的，无法根据用户后续的探索方向动态调整
2. **API 成本高**：每次生成 5 个脚本需要大量 token，DeepSeek API 调用成本较高
3. **交互体验差**：用户拿到一堆脚本后不知从何选起，缺乏引导式的探索体验
4. **内容同质化**：一次性生成多个脚本容易出现主题雷同，质量参差不齐

## 决策

将核心流程从"批量推送 5 个脚本"改为"按需拉取单个脚本"：

1. **单次生成**：每次只生成 1 个脚本，根据用户的叙事上下文（narrative context）精准匹配
2. **按需触发**：用户主动请求下一个脚本，而非系统一次性推送所有脚本
3. **队列管理**：引入 topic_queue 表管理脚本生成队列，支持优先级排序
4. **叙事驱动**：用户通过描述自己的教学故事来引导脚本生成方向

### API 变更

| 旧接口 | 新接口 | 说明 |
|--------|--------|------|
| `POST /api/scripts/generate/{id}?count=5` | `POST /api/scripts/pull` | 从批量生成改为按需拉取 |
| `GET /api/scripts/list/{id}` | `GET /api/scripts/list/{id}` | 保留，用于查看历史脚本 |

## 后果

### 正面

- **相关性提升**：每次生成都基于用户最新的叙事上下文，脚本更贴合当前需求
- **成本降低**：单次 API 调用 token 量减少约 80%（从 5 个脚本降到 1 个）
- **交互更自然**：用户像"翻书"一样逐个探索脚本，而非面对一堆选择
- **质量提升**：集中精力生成单个脚本，质量更可控
- **可扩展性**：队列机制支持未来增加优先级、定时生成等功能

### 负面

- **需要更多交互**：用户需要多次点击才能获得多个脚本
- **新增数据模型**：需要引入 story_profile 和 topic_queue 两张新表
- **迁移成本**：现有代码中 generateScripts() 方法需要重写

### 风险

- **用户习惯改变**：从"一次拿一堆"变为"一个一个拿"，需要 UI 引导
- **队列积压**：如果用户频繁拉取但不消费，队列可能积压
